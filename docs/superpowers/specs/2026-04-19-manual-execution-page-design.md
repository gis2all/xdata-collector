# 手动执行任务页设计稿

## 背景

当前 `ManualSearchPage` 已经完成了“任务包是主对象”的第一轮收口，但页面的首屏重点仍不够稳定：

- 页头操作较散，`立即执行任务` 的主语义还不够突出
- `当前任务包`、任务包操作、任务正文之间的层级还不够清晰
- 执行结果与编辑区的节奏还可以更顺手

本轮设计目标不是新增功能，而是在现有能力不变的前提下，把页面进一步收口为“执行优先”的工作台。

## 已确认结论

- 页面主语义：`手动执行任务`
- 首屏唯一主按钮：`立即执行任务`
- 任务包相关操作全部退为次级
- 采用方向 `B`：`页头主按钮 + 右侧执行轨`
- 页面仍围绕任务包草稿工作，不改变现有 task pack 行为和接口

## 设计目标

1. 用户一进页面就能明白当前在编辑并执行一个任务包草稿
2. 执行动作始终是首屏最清晰、最容易点击的操作
3. 任务包信息、任务包操作、任务正文、执行结果的层级更清楚
4. 执行后的反馈更近，但不挤压正文编辑空间

## 不在本轮范围

- 不改后端 API
- 不改 task pack schema
- 不改 `SearchSpecEditor` / `RuleSetEditor` 的字段能力
- 不新增新的运行逻辑、结果逻辑或导入逻辑
- 不顺手做全站视觉改版

## 备选方向

### A. 页头执行，线性单栏

页头放标题和主按钮，下面按线性顺序依次展示 `当前任务包`、`任务正文`、`执行结果`。

优点：

- 改动最小
- 最贴近当前实现

缺点：

- 执行虽然在首屏，但滚到正文深处后不够顺手
- 最近一次执行状态与结果反馈离编辑区域仍偏远

### B. 页头主按钮 + 右侧执行轨

页头保留唯一主按钮 `立即执行任务`，主体改为桌面双栏：左侧为编辑工作区，右侧为执行上下文轨。

优点：

- 执行主语义最清晰
- 编辑时也能稳定看到最近执行状态
- 更符合工作台式页面节奏

缺点：

- 比 A 需要更多结构调整
- 需要更认真处理桌面与窄屏断点

### C. 结果并排展示

页头仍放主按钮，主体强调“左侧任务正文，右侧执行结果”。

优点：

- 执行后结果阅读很直接

缺点：

- 未执行时右侧较空
- 任务包草稿心智不如 B 稳定

## 选型

采用 **B. 页头主按钮 + 右侧执行轨**。

原因：

- 最符合“执行优先”的明确目标
- 不会牺牲任务包草稿作为页面主对象的表达
- 能在不新增功能的前提下，显著改善操作顺手度

## 页面结构

### 1. 页头

页头只保留三项内容：

- 页面标题
- 一句用途说明
- 唯一主按钮 `立即执行任务`

其余动作不再与主按钮同层抢主语。

### 2. 主体布局

桌面端采用双栏：

- 左侧：编辑工作区
- 右侧：执行轨

窄屏时回落为上下单列，执行轨移动到编辑工作区之后、结果区之前。

### 3. 左侧编辑工作区顺序

左侧阅读顺序固定为：

1. `当前任务包`
2. `任务包操作`
3. `任务正文摘要`
4. `搜索条件`
5. `规则`

这样可以让用户先确认“当前编辑的是谁”，再决定是否换包、存包，最后进入具体正文编辑。

### 4. 右侧执行轨顺序

右侧不再承担第二套主要操作，而是承担“执行上下文”展示：

1. 当前草稿状态
2. 最近一次执行时间
3. 最近一次执行结果状态
4. `raw_total / matched_total / errors`
5. 跳转到结果区的次级入口

右侧执行轨在桌面端保持 sticky，帮助用户在长表单编辑时持续看到最近执行反馈。

### 5. 下方执行结果区

执行结果区放在双栏下方整块展开，不塞进右侧小轨道。

结果区结构保持为：

- `执行结果`
- 查询摘要 / final queries
- 原始结果
- 命中结果

命中原因、规则命中信息继续保留在结果内容中。

## 区块设计

### 当前任务包

该区块只回答“当前草稿是谁”，不承担大批量操作。

展示信息：

- 任务包名
- 说明
- `pack_name`
- 绑定状态
- 草稿状态
- 草稿来源

重点是帮助用户快速理解当前草稿是否绑定到某个本地任务包，而不是在这里堆按钮。

### 任务包操作

任务包操作单独成区，不与对象信息混在一起。

建议保持两张操作卡：

- 左卡：`载入任务包` / `从文件导入` / `导入并保存为新任务包`
- 右卡：`另存为新任务包` / `保存到当前任务包` / `删除当前任务包`

所有按钮都保持次级或危险语义，不与 `立即执行任务` 争主操作层级。

### 任务正文摘要

位于 `任务包操作` 之后、正文编辑之前。

该区块用于快速展示：

- 当前 query preview
- 当前草稿大概会搜什么

目的不是替代正文编辑，而是让用户在进入详细编辑前先快速建立上下文。

### 搜索条件

继续保留完整 `SearchSpecEditor`，但在视觉上作为主区块之一，和任务包操作分层清楚。

区块文案聚焦：

- 这里决定“搜什么”

### 规则

继续保留：

- `规则名称`
- `规则说明`
- 完整 `RuleSetEditor`

区块文案聚焦：

- 这里决定“如何筛选与分级”

## 交互原则

### 主按钮原则

- 首屏唯一主按钮固定为 `立即执行任务`
- 页面内不出现第二个等价主按钮
- 任务包相关的保存、导入、删除全部降为次级或危险动作

### 执行反馈原则

- 点击 `立即执行任务` 后，结果更新仍落在现有结果区
- 右侧执行轨只同步展示摘要，不承载完整结果阅读
- 若执行失败，右侧执行轨显示短错误摘要，详细内容仍放在结果区或错误提示中

### 空状态原则

未执行时：

- 右侧执行轨显示 `尚未执行`
- 下方结果区显示清晰空状态
- 文案明确提示：编辑好当前草稿后，点击页头 `立即执行任务`

## 响应式策略

### 桌面端

- 保持 `左侧编辑工作区 + 右侧执行轨`
- 右侧执行轨 sticky
- 下方结果区全宽展开

### 窄屏

- 回退为单列
- 顺序为：页头 -> 当前任务包 -> 任务包操作 -> 任务正文摘要 -> 搜索条件 -> 规则 -> 执行轨 -> 执行结果
- sticky 取消，避免小屏滚动负担

## 实施建议

本轮建议只做前端结构与样式调整，不改现有 API 调用方式。

允许拆出少量展示型子组件，推荐范围：

- `ManualPageHeader`
- `ManualTaskPackSummary`
- `ManualExecutionRail`

不建议做大规模组件重写，保持主要状态逻辑仍留在 `ManualSearchPage`。

## 测试建议

前端测试至少覆盖：

- 页头唯一主按钮为 `立即执行任务`
- `当前任务包`、`任务包操作`、`任务正文摘要`、`搜索条件`、`规则`、`执行结果` 全部可见
- 任务包相关操作仍存在，但不作为主按钮出现
- 桌面结构下存在执行轨摘要区
- 执行后结果区仍正确渲染 final queries / raw / matched

## 风险与注意点

- 右侧执行轨不能退化成第二套拥挤工具栏
- 结果区不能被压缩到影响阅读
- 任务包对象信息与任务包操作必须继续分层，避免重新混乱
- 如果在实现时发现页头信息过载，应优先删减次级文案，而不是增加更多提示文本

## 成功标准

满足以下条件即可视为成功：

- 用户一眼能看出“这页是在执行当前任务包草稿”
- 用户无需先理解任务包保存逻辑，也能直接找到 `立即执行任务`
- 编辑长正文时，最近一次执行反馈仍然容易看到
- 结果阅读保持完整，不因执行优先而被压缩
