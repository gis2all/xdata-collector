# XData Collector UI DESIGN.md

## 设计目标与适用范围

这份 `DESIGN.md` 是当前仓库的 UI 设计规范入口，供人和 AI 在做前端设计、页面改版、组件视觉调整和布局重构时统一遵循。

目标不是把产品做成 Apple 官网，而是借 Apple 风格的克制、清晰、秩序感，收口成一套适合数据工作台的中度 Apple 化界面语言。

这份规范优先服务以下页面：

- `自动任务`
- `手动执行任务`
- `结果浏览`

同时也约束全局导航、页面头部、卡片、表单、按钮、表格、状态提示和空状态的表达方式。

---

## 视觉气质与整体原则

### 目标气质

- 干净、克制、秩序明确
- 工具感强，但不生硬
- 信息密度中等偏高，优先保证效率
- 操作路径清晰，避免页面像“能力堆放仓库”
- 视觉重点少而准，不用过多颜色抢注意力

### 核心原则

- 一个页面先回答三件事：当前是什么、现在能做什么、下一步最重要的动作是什么
- 一个区域只承载一种主要语义，不把状态、编辑、导入、危险动作混在同一层
- 主按钮稀缺。首屏尽量只有一个真正的主操作，其余动作退成次级或辅助
- 卡片用于信息分区，不用于把内容切碎
- 表格优先扫描效率，不能为了“好看”牺牲对比和批量操作能力
- 页面先建立层级，再考虑装饰

### 明确拒绝

- 营销页式 hero
- 大面积装饰渐变
- 过度玻璃拟态
- 大量悬浮阴影和彩色描边
- 每块内容都做成抢眼卡片
- 为了“高级感”牺牲工作台效率

## 颜色角色与状态色规则

### 基础角色

建议优先使用如下角色，不要把颜色当作自由装饰素材。

```css
:root {
  --bg-page: #f5f5f7;
  --bg-surface: #ffffff;
  --bg-muted: #f2f2f4;
  --bg-subtle: #fafafc;

  --text-primary: #1d1d1f;
  --text-secondary: rgba(29, 29, 31, 0.68);
  --text-tertiary: rgba(29, 29, 31, 0.48);

  --border-default: rgba(29, 29, 31, 0.10);
  --border-strong: rgba(29, 29, 31, 0.18);

  --accent-primary: #0071e3;
  --accent-primary-hover: #0058b0;
  --accent-primary-soft: rgba(0, 113, 227, 0.10);

  --success: #1f8a65;
  --success-soft: rgba(31, 138, 101, 0.12);
  --warning: #b86a00;
  --warning-soft: rgba(184, 106, 0, 0.12);
  --danger: #d92d20;
  --danger-soft: rgba(217, 45, 32, 0.12);
  --info: #0071e3;
  --info-soft: rgba(0, 113, 227, 0.12);

  --shadow-card: 0 8px 24px rgba(15, 23, 42, 0.06);
  --shadow-popover: 0 16px 40px rgba(15, 23, 42, 0.12);
  --focus-ring: 0 0 0 3px rgba(0, 113, 227, 0.18);
}
```

### 使用规则

- 页面背景用 `--bg-page`
- 主要内容面板用 `--bg-surface`
- 次级分区、摘要块、嵌套区块用 `--bg-muted` 或 `--bg-subtle`
- 主操作、焦点态、当前激活项统一用 `--accent-primary`
- 不要把蓝色同时拿去表示“主按钮 + badge + 卡片背景 + 大标题强调”

### 状态色规则

- `success` 只用于成功结果、健康成功、启用成功等确认状态
- `warning` 只用于提醒、待确认、非阻断风险
- `danger` 只用于删除、彻底删除、错误、失败
- `info` 只用于中性提示或当前激活态
- 状态色优先以浅底 + 深字出现，不默认用纯色整块铺满

## 字体层级与文本规则

### 字体栈

优先使用系统可得字体，保持 Apple 气质，但兼容 Windows。

```css
font-family:
  "SF Pro Display",
  "SF Pro Text",
  "Segoe UI",
  "PingFang SC",
  "Microsoft YaHei",
  sans-serif;
```

### 字级层级

- 页面主标题：28-32 / 600
- 区块标题：20-22 / 600
- 子区块标题：15-17 / 600
- 正文：14-15 / 400
- 次级说明：12-13 / 400
- 表格正文：13-14 / 400
- 数值或摘要关键值：20-24 / 600

### 文本原则

- 标题负责结构，不负责抒情
- 页面说明保持短句，优先解释“这页做什么”
- 次级说明只写必要上下文，不写长教程
- 表单标签尽量直接，少用模糊词
- 一个区块里不要同时出现多个强标题层级

## 间距、圆角、边框、阴影与层级

### 间距节奏

建议以细颗粒节奏控制工作台密度：

- 4 / 8 / 12 / 16 / 20 / 24 / 32

使用建议：

- 行内控件间距：8
- 小区块内边距：12
- 普通卡片内边距：16
- 页面主区块间距：16-20
- 大模块之间：24-32

### 圆角

- 输入框、按钮、小卡片：10-12
- 大卡片、工作区面板：14-18
- pill / badge：999

### 边框与阴影

- 优先用细边框和浅层背景区分层级
- 阴影只给真正需要浮起的对象：主卡片、popover、浮层
- 普通区块不要都加重阴影

### 层级表达顺序

层级优先通过这些方式表达，按顺序从强到弱使用：

1. 信息顺序
2. 标题层级
3. 留白与分区
4. 背景深浅
5. 边框
6. 阴影
7. 强调色

## 核心组件规范

### 按钮

- 主按钮：深色或品牌蓝底，仅用于当前最重要动作
- 次级按钮：浅灰底或描边，承载普通操作
- 危险按钮：仅用于删除、彻底删除等不可逆行为
- Ghost 按钮：仅用于不应抢视觉的辅助动作

规则：

- 同一操作区尽量只有一个主按钮
- 当一个区域已经有“保存”主按钮时，导入、重置、刷新都应退为次级
- “立即运行”“保存任务”“保存到当前任务包”这类动作不要在一个视图里同时都做主按钮

### 输入框与选择器

- 默认浅色背景，细边框
- 焦点态必须有清晰蓝色 ring
- 错误态优先红色 ring + 短错误文本，不用整块大红底
- 输入控件高度统一，不要一页里出现多种高度体系

### 卡片

- 卡片只用于分区，不用于给每一条内容套壳
- 卡片内部必须有清晰标题或分区目的
- 卡片之间靠留白建立关系，不靠强彩色背景

### 表格

- 保持紧凑但可读
- 表头清晰、排序/筛选可见、列宽可控
- 行操作优先收口，不要一行堆太多抢眼按钮
- 批量操作属于表格上方的管理条，不要塞回行内

### Badge 与状态提示

- Badge 只承担状态摘要，不承担主要操作
- 尽量短：2-4 个字
- 同一视图里 badge 数量受控，避免彩色标签墙

### 工具栏

- 工具栏要分“筛选工具”和“主操作”
- 如果既有查询、筛选、批量管理，又有新建或保存，主操作必须视觉更稳、更靠右或更靠前
- 工具栏过长时，应拆成两层，而不是所有控件同排

### 分页、空状态、提示

- 分页区域保持稳定位置，靠近表格底部
- 空状态要说明“当前没有什么”以及“下一步能做什么”
- 成功提示短而明确，错误提示直接指出问题，不写泛化文案

## 页面布局原则

### 全局导航

- 左侧导航保持稳重、克制、低干扰
- 当前页高亮要清楚，但不要过度发光或高饱和
- 导航标题是产品名，不承担品牌宣传

### 页面头部

每个页面头部都优先包含：

1. 页面标题
2. 一句用途说明
3. 当前页最重要的主操作

不要把头部变成信息垃圾桶。

### 页面内部结构

页面结构尽量遵循：

1. 当前对象或当前状态
2. 主操作区
3. 主要内容区
4. 次级详情或历史信息

如果页面信息很多，优先拆成少数几个稳定大区，而不是堆很多大小接近的小卡片。

## 重点页面规则

### 自动任务

目标：让这页首先像任务管理工作台，其次才像表单集合页。

规则：

- 保留双栏布局
- 左侧继续使用表格列表，承担搜索、筛选、批量管理和任务切换
- 右侧是统一的“当前任务工作区”，默认按可编辑工作区设计，不强调 `view/edit` 切换
- 右侧阅读顺序固定为：
  - 当前任务状态摘要
  - 主保存操作
  - 调度设置
  - 当前绑定任务包
  - 任务正文摘要
  - 搜索条件
  - 规则
  - 运行统计或历史
- 顶部主保存入口必须始终可达
- 任务包相关操作要聚合，不要散落到规则编辑器内部

### 手动执行任务

目标：让用户一眼理解“当前在编辑并执行一个任务包草稿”。

规则：

- 顶部先展示当前任务包摘要和主操作
- 中部固定为“任务正文”，内部明确拆成：
  - 搜索条件
  - 规则
- 执行按钮必须在首屏清晰可见
- 执行结果区与编辑区明显分层，避免互相抢视线
- 结果摘要优先呈现查询、命中数、规则摘要，再看明细表

### 结果浏览

目标：像高效数据工作台，而不是普通内容列表页。

规则：

- 顶部先呈现当前表、筛选和主操作
- 表格是绝对主角，列控制、排序、列宽拖拽都属于辅助能力
- `raw` / `curated` 切换要明确且稳定
- 批量删除、去重、字段控制应与当前表强绑定
- 在高密度前提下保持单元格内容可扫描，避免过多嵌套卡片化

## Do / Don't

### Do

- 先做结构，再做样式
- 一屏只保留一个真正主按钮
- 用留白、边框和浅层背景建立秩序
- 让任务、任务包、结果、日志这些对象各有自己的稳定分区
- 让高频操作在首屏可达
- 让复杂页面有固定阅读顺序

### Don’t

- 不要把每个模块都做成同等强调度
- 不要把导入、保存、删除、刷新全部做成同等视觉按钮
- 不要让规则编辑器承担任务包级对象操作
- 不要为了“高级感”牺牲表格和表单效率
- 不要把 badge 当主层级工具
- 不要在一个页面里混用多套卡片和间距节奏

## 给 AI 使用的执行提示

当你在这个仓库里做前端设计或 UI 改动时，默认按以下顺序执行：

1. 先识别当前页面的主对象是谁：任务、任务包、结果、日志、设置，还是健康状态
2. 先建立阅读顺序，再决定颜色和卡片
3. 先确定唯一主操作，再安排次级和危险操作
4. 优先复用少量稳定的布局模式，不轻易发明新容器
5. 如果一个页面已经足够复杂，优先减噪，而不是继续加说明文本

默认判断标准：

- 改完后，用户应更快找到“当前是什么”“下一步点哪里”
- 页面应更像一套统一工作台，而不是几个独立 demo 页面
- 若视觉更像 Apple 官网但操作更绕，视为失败
- 若信息更清晰、操作更顺手、视觉更克制一致，则视为成功

## 与当前项目语义对齐

- `task pack = search_spec + rule_set`
- 自动任务是“调度壳 + 绑定任务包”
- 手动执行页编辑并运行的是“任务包草稿”
- 结果浏览是 `x_items_raw` / `x_items_curated` 双表工作台
- `workspace.json` 是轻量环境与 jobs registry，不是大工作区快照

这份规范描述的是目标 UI 语言。后续页面改版、组件调整和样式收口，都应以它为基线，而不是继续按单页临时判断。
